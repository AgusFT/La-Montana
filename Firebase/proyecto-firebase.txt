Objetivo
Implementar la plataforma backend mínima en Firebase para La Montaña, garantizando seguridad, consistencia de datos y automatizaciones críticas para soportar dos superficies de producto: Web Cliente y App Admin Desktop (Windows). La plataforma debe cubrir autenticación, roles, reglas de acceso, almacenamiento de archivos, notificaciones, y trazabilidad (auditoría), con un diseño robusto y mantenible.

Alcance técnico (incluye)

Modelo de datos y convenciones

Definición de colecciones, campos y convenciones (createdAt/updatedAt, ownership, estados de pedido, etc.).

Estados de pedido y transiciones válidas (state machine) compartida por web y desktop.

Roles y control de acceso (custom claims)

Roles client y admin via custom claims.

Bootstrap del primer admin mediante script local one-shot (Admin SDK).

Invitación de nuevos admins (Variante A):

Crear usuario en Firebase Auth (si no existe),

asignar claim role=admin,

generar y enviar password reset link para que el admin defina su contraseña en el primer acceso,

registrar el evento en auditoría.

Firestore Security Rules

Acceso por rol y ownership:

cliente: solo su perfil, sus pedidos, sus archivos y sus notificaciones,

admin: acceso a operación interna (pedidos, impresoras, caja, catálogo, clientes, etc.).

Validación para impedir que el cliente modifique campos operativos/admin-only.

Pruebas mínimas con Emulator Suite.

Storage (PDF/DOCX) + cuota

Layout de Storage por usuario/pedido.

Storage Rules por ownership y acceso admin controlado.

Cuota por usuario: 10MB por defecto + soporte para upgrades:

contabilidad de usedBytes vs maxBytes,

triggers en backend para sumar/restar por subida/borrado,

bloqueo de subidas que excedan plan.

Notificaciones (persistentes) + email (base para SMS)

Inbox in-app persistente en Firestore para cliente y admin responsable.

Triggers por creación/cambios de estado de pedidos.

Envío de email con plantillas mínimas versionadas.

Base preparada para integrar SMS en una fase posterior.

Auditoría (auditEvents) + observabilidad mínima

Registro append-only de acciones críticas (cambios de estado, roles, caja, precios, asignaciones, mantenimientos).

Logging básico en Functions/Run para troubleshooting.

Calidad y CI

Lint/tests + ejecución de pruebas de rules/emuladores en CI.

Fuera de alcance (por ahora)

RBAC fino por permisos (todos los admins mismo nivel).

Facturación/AFIP y pagos reales.

Analytics/BI avanzados o reporting complejo.

Integraciones con hardware de impresoras.

Dependencias

Este proyecto habilita funcionalmente a:

MVP Web Cliente (registro, pedidos, subida de archivos, checkout, notificaciones).

MVP Admin Desktop (operación de pedidos, impresoras, caja, catálogo, clientes, mapa).

Requiere definir proveedor de email (y SMS si se activa en el futuro).

Criterio de “Done” del proyecto (MVP)

Existe un esquema de roles seguro (admins no se registran públicamente) y se puede invitar admins con seteo de contraseña vía reset link.

Las Rules de Firestore/Storage impiden accesos indebidos y se validan con emuladores.

Subida de archivos funciona con cuota por usuario.

Notificaciones persistentes y email por eventos clave funcionan sin depender de que la app desktop esté abierta.

Auditoría registra acciones críticas y el CI evita regresiones básicas.
M1: Modelo de datos + Convenciones + Estados de pedido:
Objetivo
Definir la base técnica (data model) para que Web Cliente y Admin Desktop implementen funcionalidades sin inventar campos ni romper consistencia.

Incluye

Definición de colecciones/documentos mínimos (y campos con tipo):

users, orders, products, servicesPricing, printers, printerMaintenances, cashLedger, notifications, auditEvents.

Convenciones:

nombres de campos (camelCase), timestamps (createdAt, updatedAt), IDs, soft-delete .

Estados de pedido y transiciones válidas (state machine):

p.ej. created, pending_payment, paid, in_progress, ready, delivering, completed, cancelled.

Queries esperadas (para definir índices): por estado+fecha, por cliente+fecha, por impresora, etc.

No incluye

Rules ni Functions.

UI.

Entregables

Documento docs/data-model.md.

Documento docs/order-states.md con transiciones y reglas.

Checklist de índices requeridos (por query).

Criterios de aceptación

Cualquier issue de Web/Admin puede referenciar el modelo sin ambigüedades.

Estados y transiciones quedan definidos y no contradictorios.
M2 — Roles (custom claims) + Bootstrap + Invitación de Admins:
Objetivo
Tener un esquema seguro y operativo de roles (client/admin) y un flujo controlado para dar de alta administradores sin “registro público”, usando Firebase Auth + Admin SDK.

Incluye

Custom claims

Definir claim estándar: role: "admin" | "client".

Política: si no hay claim, tratar como client  o setear client explícito al registrarse.

Bootstrap (primer admin)

Script local one-shot (Admin SDK) que asigna role=admin a tu usuario inicial.

Registro en auditoría del cambio de rol.

Invitación de nuevos admins (Variante A)

Cloud Function inviteAdmin(email, displayName?) que:

crea usuario en Firebase Auth ,

asigna role=admin via custom claims,

genera password reset link (para primer seteo de contraseña),

envía email de invitación con el link,

registra auditEvent.

Refresco de token

Documentar/implementar el refresh en web/desktop para que el rol se refleje tras invitación/cambio (por ejemplo, re-login o refresh de token).

No incluye

RBAC fino por permisos (todos los admins igual).

UI compleja de “gestión de roles” (puede ser simple en desktop).

Entregables

Script local scripts/bootstrap-admin (documentado) para crear el primer admin.

Cloud Function inviteAdmin (y opcional revokeAdmin).

Documento docs/roles-and-admin-onboarding.md con:

procedimiento bootstrap,

procedimiento invitar admin,

consideraciones de seguridad.

Criterios de aceptación

No existe un camino desde la web pública para convertirse en admin.

Un admin puede invitar a otro admin y este puede setear contraseña con el link y loguearse.

Los claims son la única fuente de verdad del rol, y quedan auditados.
M3 — Firestore Rules (cliente vs admin) + pruebas con emuladores:
Objetivo
Garantizar seguridad por diseño: cliente solo accede a lo suyo; admin accede a operación interna; evitar escalamiento de privilegios.

Incluye

Rules por rol y ownership para:

users/{uid}: cliente solo su doc; admin acceso según política.

orders: cliente solo sus pedidos; admin puede leer/operar todos.

products/servicesPricing: lectura cliente; escritura admin.

printers, printerMaintenances, cashLedger, auditEvents, notifications: solo admin (salvo notifications del cliente si corresponde a su inbox).

Validación de campos “admin-only”:

evitar que cliente escriba asignaciones, notas internas, campos operativos.

Pruebas mínimas con Emulator Suite:

casos positivos/negativos (lectura ajena, escritura indebida, etc.).

Entregables

firestore.rules en repo.

Tests/scripts de rules ejecutables local y en CI.

Documento docs/security-firestore.md.

Criterios de aceptación

Cliente no puede leer ni modificar datos de otro usuario.

Cliente no puede modificar campos operativos de pedidos.

Admin puede ejecutar el flujo operativo sin workarounds.
M4: Storage: layout + Storage Rules + Cuota (10MB + upgrades):
Objetivo
Subida de PDF/DOCX segura por usuario/pedido, con aislamiento y enforcement de cuota por plan.

Incluye

Diseño de rutas de Storage (convención estable).

Storage Rules:

cliente solo sus archivos,

admin acceso controlado (p. ej. por pedido asociado).

Cuota por usuario:

users/{uid}.plan.maxBytes (default 10MB),

users/{uid}.usage.usedBytes.

Triggers:

onFinalize: sumar tamaño (transacción),

onDelete: restar tamaño.

Bloquear subidas si excede el plan (en rules/enforcement coherente).

Base para “upgrade” de plan (solo estructura; sin pagos reales).

Entregables

storage.rules en repo.

Functions de cuota.

Documento docs/storage-layout.md y docs/quota.md.

Criterios de aceptación

Un cliente no puede acceder a archivos ajenos.

No puede superar 10MB (o el plan vigente).

El contador vuelve a bajar al borrar.
M5: Notificaciones (DB) + Email (y base para SMS):
Objetivo
Notificar a cliente y admin responsable de eventos clave, con persistencia (inbox) y canales externos.

Incluye

Notificaciones in-app persistentes

Colección notifications/{uid}/items/{id} con type, payload, read, timestamps.

Triggers

Al crear pedido: notificar admins (o cola/segmentación mínima).

Al cambiar estado: notificar cliente y admin responsable.

Email

Envío de emails por estados clave (plantillas mínimas versionadas).

Base para SMS

Interfaz/abstracción para integrar proveedor más adelante (sin activar en MVP si no querés costos).

Entregables

Functions triggers de notificación (por cambios en orders).

Plantillas de email en repo.

Documento docs/notifications.md.

Criterios de aceptación

Cambios relevantes generan inbox item para cliente y/o admin responsable.

Emails se envían en los estados definidos.

No hay filtración de datos sensibles en payloads.
M6: Auditoría (auditEvents) + Observabilidad mínima + CI:
Objetivo
Trazabilidad de acciones críticas y base de calidad (tests/CI) para que el sistema sea mantenible.

Incluye

auditEvents append-only:

cambios de rol (bootstrap/invite/revoke),

cambios de estado de pedido,

asignaciones (impresora/responsable),

cambios de precios,

movimientos de caja,

altas/bajas de impresoras y mantenimientos.

Reglas de inmutabilidad:

no editar/borrar eventos (solo crear).

Observabilidad mínima:

logging estructurado en Functions.

CI:

lint/test + emuladores (rules) + tests mínimos de Functions.

Entregables

Modelo auditEvents y puntos de escritura.

Pipeline CI básico.

Documento docs/audit.md y docs/ci.md.

Criterios de aceptación

Toda acción crítica crea un auditEvent.

Clientes no pueden leer ni escribir auditoría.

CI falla si rompen rules o contratos mínimos.
M7 : Contrato de Catálogo Público (public fields) + Convenciones de publi:
Objetivo
Definir formalmente qué campos del catálogo son “publicables” (para landing) y cómo se exportan, evitando filtración de datos internos y garantizando consistencia entre Desktop (admin) y Web (público).

Incluye

Definición de “public fields” para:

products (p.ej. title, imageUrl/path, priceFrom, category, tags, isActive/published, sortOrder).

servicesPricing (p.ej. name, basePrice, unit, isActive, sortOrder).

Convenciones:

isActive / published como fuente para export.

sortOrder para orden estable del listado.

Campos internos NO publicables (costos, margen, notas internas, etc.).

Reglas de lectura pública (si aplica):

Si la landing consume solo snapshot, la lectura pública puede mantenerse mínima; documentar decisión.

Contrato del JSON exportado (schema):

generatedAt, products[], services[].

No incluye

Páginas de detalle SEO por producto (eso sería otra fase).

Motor de búsqueda avanzado.

Entregables

Documento docs/public-catalog-contract.md.

Checklist de campos por colección y ejemplo de JSON.

Recomendación de índices si hubiera lecturas públicas directas (opcional).

Criterios de aceptación

Se puede exportar catálogo sin ambigüedad (campos/tipos claros).

No se exponen campos internos en el snapshot público.

Desktop y Web consumen el mismo contrato (sin “inventar campos”).
M8 : OIDC/WIF + permisos CI (GitHub Actions → Firestore export + Hosting:
Objetivo
Habilitar CI/CD seguro y mínimo-privilegio para que GitHub Actions pueda (1) leer el catálogo publicable desde Firestore y (2) desplegar el front a Firebase Hosting, sin tokens largos.

Incluye

Configuración de Workload Identity Federation (OIDC) entre GitHub y GCP/Firebase:

Provider + Service Account dedicado a CI.

Permisos mínimos:

Lectura de Firestore necesaria para export del catálogo (solo lo requerido).

Permiso para deploy a Firebase Hosting.

Gestión de secretos/variables:

FIREBASE_PROJECT_ID como variable.

Secrets para provider/SA (si aplica) evitando credenciales permanentes.

Documentación operativa:

docs/ci-web-deploy.md con setup paso a paso y troubleshooting.

No incluye

Automatizaciones “por evento de Firestore” (publicación automática al editar catálogo).

Multi-entorno completo (dev/staging/prod) si no lo querés aún.

Entregables

WIF/OIDC configurado.

Service Account CI con roles mínimos.

Workflow de GitHub Actions funcionando end-to-end.

Criterios de aceptación

El pipeline despliega sin usar tokens permanentes.

Si el SA pierde permisos, el workflow falla (principio de mínimo privilegio).

Queda documentado el procedimiento y cómo rotar/ajustar permisos.
